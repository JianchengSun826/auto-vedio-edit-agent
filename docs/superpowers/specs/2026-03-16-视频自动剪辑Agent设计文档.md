# 视频自动剪辑 Agent — 设计文档

**日期**：2026-03-16
**状态**：已确认

---

## 概述

一个 AI 驱动的视频自动剪辑 Agent，接受用户的自然语言剪辑需求，处理本地视频文件，生成候选剪辑方案，经人工审核确认后导出适配多个社交媒体平台的成品视频。

---

## 目标

- 接受用户自然语言输入的剪辑需求
- 支持三种剪辑模式：精华片段提取、素材拼接、社媒内容生产
- 处理本地视频文件（架构可扩展，支持未来接入云存储）
- 使用 STT + LLM 理解用户意图，生成结构化剪辑方案
- 导出前必须经过人工审核确认
- 导出平台：抖音/TikTok、B站、YouTube、微信视频号
- 控制成本：中等用量目标 < $5/月（约100个视频/月）

---

## 非目标（v1 范围外）

- 实时/直播视频处理
- 视觉 AI（人脸检测、场景理解）
- 移动端或原生桌面客户端
- 多用户 / SaaS 平台

## 模块设计

### 存储抽象层

所有模块只依赖 `StorageBackend` 抽象接口。未来新增云存储（S3、GCS）只需实现该接口，无需修改任何上层代码。

```python
# storage/base.py
class StorageBackend(ABC):
    def read(self, path: str) -> Path: ...    # 返回本地临时路径
    def write(self, local_path: Path, dest: str): ...
    def list(self, prefix: str) -> list[str]: ...
```

- **v1**：`LocalStorage` — 读写本地文件系统
- **未来**：`S3Storage` — 下载到临时目录处理，写入时上传
- 通过 `settings.py` 切换：`STORAGE_BACKEND = "local"` → `"s3"`

### 输入管理器

- 接受本地文件路径
- 验证视频格式（mp4、mov、avi、mkv）
- 支持批量输入（多文件队列）
- 通过 `StorageBackend.read()` 代理读取

### STT 转录模块（`processing/transcriber.py`）

- **工具**：faster-whisper
- **默认模型**：`medium`（速度与质量均衡）
- **可配置**：`tiny` / `base` / `small` / `medium` / `large-v3`
- 内置 VAD（语音活动检测），自动跳过静音段
- 输出格式：`List[Segment]`，每条含 `{start, end, text}`
- 超过 2 小时的视频：拆分为 30 分钟块，结果合并
- 无语音时优雅降级（返回空转录，仍可用时间段/静音规则）

**各模型性能参考（CPU 环境）：**

| 模型 | 参数量 | CPU 速度 | 准确率 |
|------|--------|---------|--------|
| tiny | 39M | ~32x 实时 | 一般 |
| base | 74M | ~16x 实时 | 较低 |
| small | 244M | ~6x 实时 | 良好 |
| medium | 769M | ~2x 实时 | 很好（推荐） |
| large-v3 | 1550M | ~1x 实时 | 最佳 |

### LLM 意图解析模块（`agent/intent_parser.py`）

- **工具**：Claude Haiku（`claude-haiku-4-5`）
- 输入：用户自然语言需求 + 转录文本 JSON
- 输出：结构化 `EditPlan`（Pydantic 模型，JSON 输出模式）
- JSON 解析失败时重试一次，仍失败则返回原始规则
- 系统 prompt 严格定义 JSON Schema

**EditPlan 数据结构：**
```json
{
  "mode": "highlight_extraction | material_assembly | social_media",
  "rules": [
    {
      "type": "keyword_match | time_range | silence_cut | min_duration",
      "keywords": ["竞品", "价格"],
      "padding_before_sec": 3,
      "padding_after_sec": 5,
      "min_duration_sec": 10
    }
  ],
  "output_formats": [
    {
      "platform": "douyin | bilibili | youtube | wechat",
      "ratio": "9:16 | 16:9 | 1:1",
      "max_duration_sec": 60,
      "resolution": "1080p"
    }
  ],
  "segment_count_hint": 3
}
```

### 规则执行引擎（`agent/rule_engine.py`）

- 消费 `EditPlan`，在转录文本上执行规则匹配
- `keyword_match`：模糊匹配关键词 → 提取时间窗口（含前后 padding）
- `time_range`：直接按时间码提取片段
- `silence_cut`：调用 FFmpeg `silencedetect` filter，去除静音段
- `min_duration`：过滤掉时长低于阈值的片段
- 输出：`List[CandidateSegment]`，含 `{id, start, end, text_preview, confidence_score}`

### FFmpeg 工具库（`processing/ffmpeg_utils.py`）

- 所有 FFmpeg 调用通过 `subprocess` 直接执行（不使用 ffmpeg-python 封装）
- **剪切**：流复制（`-c copy`）— 无损、极快
- **重新编码**：仅在格式转换时触发
- **拼接**：FFmpeg concat demuxer（同格式文件无需重编码）
- **静音检测**：`silencedetect` 音频滤镜，解析 stderr 输出
- **平台导出**：画面比例裁切（居中裁）、分辨率缩放、各平台编码预设

### 人工审核界面（`app/main.py` — Gradio 5.x）

- 候选片段列表：序号、时间范围、转录文本预览、置信度分数
- 内嵌视频播放器：点击片段即跳转预览
- 每条片段的操作：勾选/取消、时间码微调
- 素材拼接模式支持拖拽排序
- "批准并导出"按钮触发导出流水线
- "重新生成"按钮以修改后的指令重新调用 LLM

### 导出模块（`processing/exporter.py`）

各平台规格：

| 平台 | 比例 | 最大时长 | 分辨率 | 备注 |
|------|------|---------|--------|------|
| 抖音/TikTok | 9:16 | 60秒（超出自动分段） | 1080×1920 | H.264 |
| B站 | 16:9 | 无限制 | 1080p / 4K | H.264 / HEVC |
| YouTube | 16:9 | 无限制 | 1080p / 4K | H.264 |
| 微信视频号 | 9:16 | 10分钟 | 1080×1920 | H.264 |

- 批量导出：多平台并行处理（Celery group）
- 自动分段：超出平台最大时长时自动拆分，按序号命名
- 输出命名规则：`{原文件名}_{平台}_{序号}.mp4`

### 异步任务队列（`tasks/celery_tasks.py`）

- **Broker + Backend**：Valkey（Redis 兼容，BSD 许可证）
- 转录任务：长耗时，`acks_late=True`，`max_tasks_per_child=10`
- 导出任务：按平台拆分，以 Celery group 并行运行
- 任务状态通过轮询推送至 Gradio UI

## 三种剪辑模式

### 模式一：精华片段提取

- 用户输入：`"提取所有提到竞品价格的片段，前后各保留5秒"`
- 流程：转录 → LLM 识别关键词规则 → 规则引擎匹配时间码 → 生成候选列表

### 模式二：素材拼接

- 用户输入：`"按脚本顺序拼接：开场30秒用第一个视频，产品演示用第二个视频2-4分钟"`
- 流程：LLM 解析脚本结构 → 在多个输入文件中匹配对应片段 → FFmpeg concat

### 模式三：社媒内容生产

- 用户输入：`"把这个采访剪成3条抖音短视频，每条不超过60秒，内容完整"`
- 流程：转录 → LLM 按语义分段 → 自动裁 9:16 → 生成3个候选方案供审核

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| 视频无语音 | 跳过 STT，仅提供时间段/静音规则 |
| LLM 输出非法 JSON | 重试一次；仍失败则返回解析错误及原始输出 |
| FFmpeg 转码失败 | 捕获 stderr，将具体错误行展示给用户 |
| Celery worker 崩溃 | `acks_late=True` — 任务自动重新入队，不丢失 |
| 视频超过 2 小时 | 拆分为 30 分钟块转录，结果合并后处理 |
| 不支持的视频格式 | 在输入阶段校验，立即返回明确错误信息 |
